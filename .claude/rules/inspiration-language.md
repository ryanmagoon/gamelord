# Inspiration Language

When describing features in PRs, commit messages, issues, or code comments, **describe the behavior, not the source.**

## Rules

- **Don't name other products as the inspiration.** Instead of "Nintendo Switch-style page transitions" write "momentum-based page transitions with spring easing." Instead of "Netflix shelf layout" write "horizontal scroll rows grouped by category." The behavior speaks for itself.
- **Describe the interaction, not the brand.** Focus on what the user sees and feels: the motion curve, the layout pattern, the timing, the feedback. If the description only makes sense when you know the reference, it's too referential.
- **References in conversation are fine.** In chat with the user, referencing other products for alignment ("like how Steam Big Picture does it") is useful shorthand. But once the feature is defined, the written spec, PR, and code should stand on their own without namedropping.
